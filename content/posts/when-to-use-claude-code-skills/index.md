클로드 코드에 스킬 기능이 추가됐다. 처음엔 헷갈렸다. 이미 에이전트도 있고 슬래시 커맨드도 있는데, 스킬은 언제 쓰라는 건지 감이 안 왔다.

며칠 사용해보니 왜 호들갑을 떠는지 알 것 같다. 스킬은 기존에 아쉬웠던 클로드코드 기능등을 많이 보완해주고 대체한다.

## 스킬은 '독립적인 기능'일 때

스킬은 독립적으로 실행할 수 있는 작업에 쓰면 된다. 프롬프트가 포함된 하나의 '기능'이라고 보면 쉽다.

예를 들어 블로그 포스트를 작성할 때를 생각해보자. 이미지를 최적화하는 작업이 매번 필요하다고 치자. 이미지 크기를 줄이고, WebP로 변환하고, alt 텍스트도 추가해야 한다.

이런 작업을 스킬로 만들면 편하다. 클로드 코드에게 이렇게 요청하면 된다:

```text
스킬 공식 문서를 참조해서 이미지 최적화 스킬을 만들어줘.

기능:
- 메인 이미지: 1200px로 리사이징하고 WebP로 변환
- 썸네일: 300px로 리사이징
- 상품 이미지: 800px로 리사이징하고 품질 90%로 설정
- 모든 이미지에 구체적인 alt 텍스트 추가
```

그럼 클로드가 `.claude/skills/optimize-images/` 폴더를 만들고 `SKILL.md` 파일에 상세한 지침을 담아준다.

이후엔 "이미지 좀 최적화해줘"라고만 하면 클로드가 알아서 이 스킬을 찾아서 적용한다.

코딩할 때도 마찬가지다. API 문서를 자동으로 생성하는 작업이 반복된다면 스킬로 만들면 된다. 함수 주석을 읽어서 Swagger 문서를 생성하는 프롬프트를 스킬에 담아두는 식이다.

## 슬래시 커맨드와 뭐가 다를까?

슬래시 커맨드도 있는데 왜 스킬이 따로 나왔을까? 써보니까 스킬이 슬래시 커맨드의 고급 버전 같았다.

가장 큰 차이는 **누가 호출하느냐**다.

슬래시 커맨드는 내가 `/명령어`를 직접 입력해야 한다. 스킬은 클로드가 알아서 판단해서 사용한다. 내 요청을 보고 "아, 이건 이미지 최적화 스킬을 써야겠네"하고 자동으로 적용하는 거다.

여기에 더해서 **스킬은 폴더 구조를 가질 수 있다**. 슬래시 커맨드는 단순한 텍스트 파일 하나지만, 스킬은 여러 파일과 코드를 함께 담을 수 있다.

아까 예시로 든 이미지 최적화를 다시 보자.

**슬래시 커맨드라면:**

```text
.claude/commands/optimize-images.md

"이미지 크기 줄이고, WebP로 변환하고, alt 텍스트 추가해줘"
```

프롬프트 파일 하나에 모든 지시를 때려넣어야 한다.

**스킬이라면:**

```text
.claude/skills/optimize-images/
├── SKILL.md            # 메인 스킬 정의 (필수)
├── hero.md             # 메인 이미지 최적화 가이드
├── thumbnail.md        # 썸네일 최적화 가이드
├── product.md          # 상품 이미지 최적화 가이드
└── scripts/
    └── optimize.sh     # 실제 최적화 스크립트
```

이미지 용도별로 가이드 문서를 구조화해서 정리할 수 있다. SKILL.md에서 hero.md, thumbnail.md, product.md를 참조하는 식이다.

게다가 실제 최적화 스크립트도 함께 넣어둘 수 있다. 클로드가 이 스크립트를 참고하거나 실행해서 작업한다.

대부분의 슬래시 커맨드는 스킬로 대체 가능하다. 오히려 스킬로 만드는 게 더 낫다. 프롬프트만 있는 간단한 경우라도 나중에 코드를 추가할 수 있으니까.

## CLAUDE.md는 '규칙'일 때

그럼 CLAUDE.md는 언제 쓸까? 규칙을 정의할 때다.

"중복 코드를 만들지 마", "타입스크립트는 strict 모드로 작성해", "커밋 메시지는 한글로 써" 같은 것들 말이다.

스킬과 다른 점은 이건 항상 적용되는 규칙이라는 거다. 내가 따로 호출할 필요 없이 클로드가 알아서 따라야 하는 것들이다.

블로그로 치면 "포스트는 항상 퇴고를 2번 이상 해라", "이미지 alt 텍스트는 구체적으로 써라" 같은 규칙을 CLAUDE.md에 넣어둔다. 그럼 클로드가 포스트 쓸 때마다 자동으로 이 규칙을 따른다.

## 에이전트는 '역할놀이'가 필요할 때

에이전트는 특정 작업에 특화된 AI가 필요할 때 쓴다.

블로그 예시로 보면 이해가 빠르다. 재테크 포스트를 쓰는 에이전트와 생활 정보 포스트를 쓰는 에이전트를 따로 만드는 거다.

재테크 에이전트는 "나는 재테크 전문가야. 주식, 부동산, 세금 관련 내용을 쓸 때는 구체적인 수치와 근거를 제시해야 해"라는 역할을 갖는다.

생활 정보 에이전트는 "나는 일상 팁 전문가야. 실생활에 바로 적용 가능한 꿀팁을 쉽게 설명해야 해"라는 역할을 갖는다.

똑같은 작업(포스트 작성)이어도 역할에 따라 접근 방식이 달라진다. 이럴 때 에이전트를 쓴다.

개발로 치면 "백엔드 API 개발자", "프론트엔드 UI 개발자" 같은 역할을 나눌 때 쓸 수 있다. 같은 기능 개발이어도 백엔드 에이전트는 성능과 확장성을, 프론트엔드 에이전트는 UX와 접근성을 중점으로 본다.

## 에이전트끼리 공유하는 기능은 스킬로

블로그 포스트 작성 에이전트랑 SNS 콘텐츠 작성 에이전트가 있다고 치자. 둘 다 이미지 최적화가 필요하다.

이 이미지 최적화 작업을 각 에이전트마다 프롬프트로 넣으면 중복이다. 이럴 때 이미지 최적화를 스킬로 만들어두면 된다.

그럼 어떤 에이전트를 쓰든 클로드가 이미지 작업이 필요할 때 자동으로 최적화 스킬을 사용한다.

쇼핑몰 개발로 보자면 "상품 페이지 개발 에이전트", "이벤트 페이지 개발 에이전트"가 있을 때, 둘 다 상품 이미지를 최적화해야 한다. 이미지 최적화 스킬을 만들어두면 어느 에이전트든 필요할 때 자동으로 활용한다.

## 정리하면

스킬은 처음엔 왜 만들었지? 싶었는데 세상에서 제일 많이 쓰고있다.

- **스킬**: 독립적인 기능이 필요할 때
- **슬래시 커맨드**: 시스템적인 기능이 필요할 때
- **CLAUDE.md**: 규칙을 명시할 때 (예: 코딩 컨벤션, 커밋 규칙)
- **에이전트**: 역할 놀이가 필요할 때 (예: 재테크 포스트 작성, 백엔드 개발)

나름대로 저 구분으로 잘 사용하고 있다. 하지만 본질은 컨텍스트 엔지니어링일 뿐이다. 여러 사용 후기를 보면 사용 방법은 완전히 자리잡진 못했으니 참고만 하자. 예를 들면 에이전트는 토큰을 너무 많이 먹어서, 에이전트가 가져야 할 컨텍스트를 스킬로 푸는 경우도 봤다. (ex. 이벤트 페이지 가이드라인을 스킬로 생성)

너무 혼란스러운 사람에게 조금이라도 도움이 되면 좋겠다. 그럼 이만.
